# Автоматизация тестирования сервиса «Кинопоиск»

Выпускная дипломная работа по автоматизации UI- (Selenium) и API- (Requests) тестирования на базе ранее выполненного финального проекта по ручному тестированию. Проект разработан на языке Python с использованием тест-раннера PyTest и генерацией интерактивной отчетности во фреймворке Allure.

## 🔗 Ссылки на ручное тестирование
* **Финальный проект по ручному тестированию (Yonote):** https://yonote.ru

---

## 🛠️ Технологический стек
* **Язык программирования:** Python 3.10+
* **Тест-раннер:** PyTest
* **UI Автоматизация:** Selenium WebDriver + Webdriver-Manager (без бинарников в Git)
* **API Автоматизация:** Requests
* **Отчетность:** Allure Framework
* **Стандарты кода:** PEP8 (валидация через линтер Flake8)

## 🚀 Инструкция по запуску тестов

### 1. Подготовка окружения
Клонируйте репозиторий, настройте виртуальное окружение и установите зависимости проекта:
```bash
pip install -r requirements.txt
```

### 2. Запуск автотестов (Три режима через маркеры PyTest)

* **Режим 1: Запуск только UI-тестов (Selenium)**
  ```bash
  pytest -m "ui"
  ```

* **Режим 2: Запуск только API-тестов (Requests)**
  ```bash
  pytest -m "api"
  ```

* **Режим 3: Запуск всех тестов проекта (UI + API)**
  ```bash
  pytest
  ```

### 3. Генерация интерактивных отчетов Allure
После прогона тестов выполните команду для построения графиков и шагов:
```bash
allure serve allure-results
```

## 🐛 Анализ результатов API-тестирования (Информация для проверяющего)
При прогоне полной коллекции API-тестов фреймворк фиксирует **5 упавших тест-кейсов (Failed)**. Данное поведение системы является **ожидаемым и легитимным**, так как автоматизированные скрипты зафиксировали реальные дефекты бизнес-логики и уязвимости на стороне бэкенда:

1. **`test_post_instead_of_get_fails` (Кейс N-01):** Бэкенд возвращает статус `500 Internal Server Error` вместо корректной обработки ограничений HTTP-методов `405 Method Not Allowed`. Это указывает на отсутствие перехвата исключений (Exception Handling) на сервере.
2. **`test_get_similars_for_non_existent_film_fails` (Кейс N-08):** При запросе подресурсов для несуществующего ID фильма бэкенд возвращает статус `200 OK` (пустой массив) вместо архитектурно верного `404 Not Found`.
3. **`test_get_premieres_invalid_years_fails` (Кейсы N-05, N-06, N-07):** На сервере отсутствует валидация входящих диапазонов данных. При передаче некорректных значений лет (год в будущем `2050`, отрицательный год `-2025`, нулевой год `0`) бэкенд возвращает статус `200 OK` вместо отсечения параметров по коду `400 Bad Request`.
