# # Автоматизация тестирования сервиса «Кинопоиск»

Выпускная дипломная работа по автоматизации UI- (Selenium) и API- (Requests) 
тестирования на базе ранее выполненного финального проекта по ручному 
тестированию. Проект разработан на языке Python с использованием 
тест-раннера PyTest и генерацией интерактивной отчетности Allure.

## 🔗 Ссылки на ручное тестирование

## 🚀 Инструкция по запуску тестов

### 1. Подготовка окружения и безопасность
Клонируйте репозиторий, настройте виртуальное окружение и установите пакеты:
```bash
pip install -r requirements.txt
```
Для безопасного хранения токенов создайте в корне проекта файл `.env` и 
добавьте туда ваш секретный ключ API Кинопоиска (файл скрыт в `.gitignore`):
```text
API_KEY=ваш_секретный_токен_из_postman
```

### 2. Запуск автотестов (Три режима через маркеры PyTest)

* **Режим 1: Запуск только UI-тестов (Selenium)**
  ```bash
  pytest -m "ui"
  ```

* **Режим 2: Запуск только API-тестов (Requests)**
  ```bash
  pytest -m "api"
  ```

* **Режим 3: Запуск всех тестов проекта (UI + API)**
  ```bash
  pytest
  ```

### 3. Работа с отчетами Allure
Построение статического отчета в фоне и его автоматический запуск в браузере:
```bash
allure serve allure-results
```
Альтернативный вариант — генерация чистой автономной папки с отчетом и 
его ручное открытие:
```bash
allure generate allure-results --clean
allure open
```

## 🐛 Анализ результатов тестирования (Информация для проверяющего)

### 1. Результаты API-тестирования
Известные баги бэкенда успешно локализованы, оформлены в виде задач во 
вкладке **GitHub Issues** текущего репозитория и помечены в коде маркёром 
`@pytest.mark.xfail` (Инженерная этика соблюдена, тестов не создают шума):
* **`test_post_instead_of_get_fails` (Баг BUG-001 / Issue #1):** Бэкенд 
отдает статус `500` вместо обработки ограничений методов `405`.
* **`test_get_similars_for_non_existent_film_fails` (Баг BUG-003 / Issue #3):** 
На запрос к подресурсу несуществующего ID бэкенд возвращает `200` вместо `404`.
* **`test_get_premieres_invalid_years_fails` (Баг BUG-002 / Issue #2):** 
Отсутствует валидация входящих диапазонов лет. На невалидные значения лет 
бэкенд отдает успешный статус `200 OK` вместо `400 Bad Request`.

### 2. Результаты UI-тестирования
Локаторы и сценарии полностью обновлены под актуальную вёрстку промышленного 
контура с использованием устойчивых относительных путей XPath и CSS-селекторов. 
Базовые таймауты явных ожиданий расширены до 30 секунд для компенсации сетевых 
задержек и обеспечения возможности ручного прохождения капчи при локальном 
прогоне. Сборка успешно проходит на 100% со статусом `PASSED`.
